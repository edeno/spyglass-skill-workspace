# Shortening `sort_interval` on a `SpikeSortingRecordingSelection` row

Short answer: it depends on **how** you "shorten" it. The cascade is very different between editing the underlying interval in place vs. pointing the selection at a new (shorter) interval. The old curation can sometimes be reused as an artifact, but it stops being a faithful representation of the new selection in either case. Below I separate the two scenarios and walk the v1 chain.

> Caveat on terminology. In Spyglass **v1** there is no field literally named `sort_interval` on `SpikeSortingRecordingSelection`; the selection's PK includes `recording_id` (UUID) plus an `interval_list_name` (FK to `common.IntervalList`). The actual time window lives in `IntervalList.valid_times`. (Spyglass v0 has a separate `SortInterval` table — `spikesorting/v0/...` — with a `sort_interval_name` PK and a `sort_interval` blob.) I'll address both possible meanings.

## Scenario A: Edit `IntervalList.valid_times` in place (same `interval_list_name`)

You leave the selection row alone but mutate the IntervalList blob it points at, halving its duration.

What changes structurally:

- Nothing in the **schema** changes. `recording_id` is unchanged. The row in `SpikeSortingRecordingSelection`, `SpikeSortingRecording`, `ArtifactDetectionSelection`, `ArtifactDetection`, `SpikeSortingSelection`, `SpikeSorting`, and `CurationV1` all still exist and are still keyed the same way.
- Every cached **content** that was computed against the old (longer) interval is now silently stale:
  - `SpikeSortingRecording.populate` writes a preprocessed recording to an analysis NWB scoped to that interval (`spikesorting/v1/recording.py` populate handler that builds an SI recording slice). Halving the interval does not retrigger populate — the file on disk still represents the old window.
  - `ArtifactDetection` was computed over the old window, so its detected artifact intervals reference samples outside the new window.
  - `SpikeSorting`'s detected spike trains were computed over the old window — half (or more) of the spikes can fall outside the new interval.
  - `CurationV1` rows (and any `MetricCuration`, `FigURLCuration`, `BurstPair`, `SpikeSortingOutput.CurationV1` merge rows) all still resolve, but their content was computed against the longer recording.

So nothing downstream is *deleted* or *invalidated by DataJoint*; the integrity check is on FKs, not blob content. This is the dangerous case: queries keep working, results look plausible, and the analysis is wrong.

What you should do:

1. Treat the in-place edit of `valid_times` as a destructive content change and follow the inspect-before-destroy loop (SKILL.md Core Directive #1; see `references/destructive_operations.md`). Don't mutate it without recomputing downstream.
2. Recompute the chain: `SpikeSortingRecording → ArtifactDetection → SpikeSorting → CurationV1 → SpikeSortingOutput`. In practice this means deleting the cached rows and re-running `populate()`, because populate is keyed on the row's UUID, not on the blob it indirectly references.
3. The old curation labels (`reject`/`noise`/`mua`/`accept`) and merge groups are *informational only* under the new window — the unit IDs in the new sort almost certainly will not line up with the old sort's unit IDs (clusters depend on the data). You cannot mechanically re-attach them.

## Scenario B: Point the selection at a new (shorter) `interval_list_name`

Insert a new row in `IntervalList` with the halved window, then call `SpikeSortingRecordingSelection.insert_selection({..., "interval_list_name": new_name, ...})`. This is the clean path.

What changes:

- A **new `recording_id` UUID** is generated by `insert_selection` (`spikesorting/v1/recording.py:176-182`). The original row is untouched.
- The original row's downstream chain (`SpikeSortingRecording`, `ArtifactDetectionSelection`/`ArtifactDetection`, `SpikeSortingSelection`/`SpikeSorting`, `CurationV1`, `MetricCuration`, `FigURLCuration`, `BurstPair`, `SpikeSortingOutput.CurationV1`) is **not modified** — it stays bound to the old `recording_id` and the old (longer) interval.
- Nothing exists yet for the new `recording_id`; you have to populate `SpikeSortingRecording`, then re-run artifact detection, sorting, and curation against the new key.

Cascade summary for the **new** branch:

```
new IntervalList row  ─┐
                       ├─ SpikeSortingRecordingSelection (new recording_id)
SortGroup (unchanged) ─┘
                          ↓ populate
                       SpikeSortingRecording (new analysis NWB, shorter)
                          ↓
                       ArtifactDetectionSelection → ArtifactDetection
                          ↓
                       SpikeSortingSelection → SpikeSorting (new sort)
                          ↓
                       CurationV1 (new curation_id)
                          ↓
                       SpikeSortingOutput.CurationV1 (new merge_id)
```

The old branch and the new branch coexist; downstream consumers route through `SpikeSortingOutput` by `merge_id`, so existing analyses pinned to the old `merge_id` keep working unchanged.

## Can the old curation be reused?

Mostly no, with one nuance.

**Why it generally can't be reused as labels:**

- Curation labels in v1 attach to **`unit_id`s within a specific `sorting_id`** (`CurationV1`'s key is `sorting_id, curation_id`; labels live in the analysis NWB units table — `references/spikesorting_v1_pipeline.md` Step 4). Re-running the sorter on a different time window produces a **different `sorting_id`** with a **different unit-ID space**. Cluster identities depend on which spikes are present; a unit that was unit 7 in the old sort is not necessarily unit 7 in the new sort, and may not exist at all. Mechanically copying the labels dict is not a sound operation.
- Merge groups have the same problem: they're sets of `unit_id`s within the old sort.

**The only safe re-use paths:**

1. **Re-curate from scratch on the new sort.** Run `MetricCuration` → review in FigURL → insert a new `CurationV1` with fresh labels. Use the old curation as a *reference* (which physical channels held interneurons, which sort groups had a lot of noise) but not as input.
2. **Subset, don't re-sort.** If your goal is "I want my downstream analysis to use only half the time," consider not changing the sort at all — keep the existing sort over the long interval and instead restrict at the *analysis* layer (e.g., pass a shorter time window to `get_spike_indicator(key, time)` / `get_firing_rate(key, time, ...)` on `SpikeSortingOutput`, or restrict the `IntervalList` you pass to decoding/MUA/ripple). This preserves the old curation exactly as-is and avoids re-running the pipeline. It's usually the right answer when "shorten the sort_interval" really means "look at less time."
3. **Edge case: the new window is a strict subset and you trust unit identities.** If the new interval is an exact subset of the old one *and* the sorter is deterministic given a subset of samples (rarely true — most sorters are sensitive to data length and noise statistics), you might in principle reuse labels. In practice I would not; verify against a held-out check before relying on this.

## Verifying which downstream rows actually exist

Before deciding what to recompute or delete, enumerate the existing chain so you don't act on assumptions. From the source/runtime graph guidance in `references/feedback_loops.md`:

```python
# What rows currently descend from the recording row?
rec_key = {"recording_id": rid}
print(len(SpikeSortingRecording & rec_key))
print(len(ArtifactDetectionSelection & rec_key))
print(len(SpikeSortingSelection & rec_key))

# Sorts and curations downstream
sort_keys = (SpikeSortingSelection & rec_key).fetch("KEY", as_dict=True)
for sk in sort_keys:
    print(len(SpikeSorting & sk),
          len(CurationV1 & {"sorting_id": sk["sorting_id"]}))

# Merge entries (merge_id is the only PK on the master)
from spyglass.spikesorting.spikesorting_merge import SpikeSortingOutput
print(SpikeSortingOutput.merge_restrict({"recording_id": rid}))
```

This tells you what's at risk and what (if anything) you need to delete via `cautious_delete`/`merge_delete` (with the inspect-before-destroy loop) before recomputing. Do not skip the inspect step — `.delete()` cascades, and the database holds irreplaceable analysis state (SKILL.md Core Directive #1).

## TL;DR

- If you mutate `IntervalList.valid_times` in place, **every downstream row keeps existing FK-wise but its cached content is now stale** — recording, artifact, sort, curation, metric curation, burst pair, and the `SpikeSortingOutput.CurationV1` merge row all need recomputation. No DataJoint error will tell you this.
- If you instead insert a new (shorter) interval and call `insert_selection` again, you get a **brand-new `recording_id` and a parallel chain**; the original chain is untouched and old downstream consumers (decoding, ripple, MUA pinned to the old `merge_id`) keep working.
- Reusing the **old curation labels** on the new sort is generally not sound, because cluster identities and `unit_id`s change with the input data. Re-curate the new sort, or — more often the right move — keep the old sort and just restrict the analysis layer to a shorter time window.
