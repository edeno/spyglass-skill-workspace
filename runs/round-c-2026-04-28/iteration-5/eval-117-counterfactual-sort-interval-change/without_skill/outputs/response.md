# Shortening `sort_interval` on a `SpikeSortingRecordingSelection` row

## Short answer

If you change the `sort_interval` (via the `IntervalList` it points to) for one
`SpikeSortingRecordingSelection` row, **everything downstream of that row in the
v1 spike sorting pipeline is invalidated and must be recomputed**. The old
`CurationV1` cannot be reused on the new (shorter) recording — the spike times,
unit IDs, and waveforms are tied to the original recording's time range and the
sort it produced. You can keep the *labels* (a curation is "just" a manifest of
unit_ids → labels/merge groups), but you have to re-derive a sort on the new
interval and then re-apply / re-curate against it.

Below is the chain and what changes, plus the practical paths forward.

---

## How `sort_interval` flows through the v1 pipeline

In `spyglass.spikesorting.v1`, the dependency chain is roughly:

```
IntervalList  (the sort_interval entry — keyed by nwb_file_name + interval_list_name)
   |
   v
SpikeSortingRecordingSelection   <-- you are editing this row's sort_interval ref
   |
   v
SpikeSortingRecording            <-- preprocessed, filtered, time-sliced recording (NWB on disk)
   |
   v
SpikeSortingSelection
   |
   v
SpikeSorting                     <-- raw sorter output (KiloSort/MountainSort/etc.)
   |
   v
CurationV1                       <-- curation_id 0 = raw sort; later ids layer edits
   |
   +--> MetricCurationSelection -> MetricCuration -> MetricCurationLabels/MergeGroups
   |       (auto-metrics: snr, isi_violation, nn_noise_overlap, presence_ratio, ...)
   |
   +--> FigURLCurationSelection -> FigURLCuration -> sortingview labels/merge groups
   |       (manual curation in sortingview, written back as a new CurationV1)
   |
   +--> SpikeSortingOutput (merge table)  ---> downstream:
            ClusterlessDecodingV1 / SortedSpikesGroup / SortedSpikesDecodingV1,
            BurstPair, UnitInclusionParameters, etc.
```

A few key facts that drive what is "stale":

1. `SpikeSortingRecording` *materializes the sliced + preprocessed recording* to
   disk as an NWB file, with the `sort_interval` baked in (start/stop samples,
   filtering, referencing). Its hash/key includes the interval list name.
2. `SpikeSorting` runs the sorter on that materialized recording, so its outputs
   (spike times, templates, unit_ids) are valid only for that interval.
3. `CurationV1` rows reference a specific `(sorting_id, parent_curation_id)` —
   curation_id 0 is the raw sort, subsequent ids are edits. They store labels
   and merge groups by `unit_id`, and those `unit_id`s are defined by the sort
   above them.
4. `SpikeSortingOutput` is a merge table; each entry points at a specific
   `CurationV1` row.

## What changes if you halve `sort_interval`

Assuming you've actually changed the interval (either by editing the
`IntervalList` row it points to, or repointing the selection at a new
`interval_list_name` — see "How to even do this" below), the following rows
become stale and have to be re-populated:

| Table                                  | Status after the change                                                                                                  |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| `SpikeSortingRecordingSelection` (row) | Logically pointing at a different recording window. Hash will reflect the new interval.                                  |
| `SpikeSortingRecording`                | Stale. The sliced/filtered NWB on disk corresponds to the old interval. Must repopulate.                                  |
| `SpikeSortingSelection`                | Stale (FK depends on the recording).                                                                                      |
| `SpikeSorting`                         | Stale. New sorter run on a shorter recording will give different unit counts, different `unit_id`s, different templates. |
| `CurationV1` (curation_id 0)           | Stale. It's the raw sort, regenerated by the new `SpikeSorting`.                                                          |
| `CurationV1` (curation_id > 0, your manual curation) | **Cannot be auto-reused** — the `unit_id`s it references were defined by the old sort. See below.        |
| `MetricCurationSelection` / `MetricCuration` | Stale; metrics are computed against the sort.                                                                       |
| `FigURLCurationSelection` / `FigURLCuration` | Stale; sortingview view is built from the sort + recording.                                                          |
| `SpikeSortingOutput` merge entry       | Points at a now-stale `CurationV1`. Must insert a new entry once you've re-curated.                                      |
| `SortedSpikesGroup`, `BurstPair`, decoding (`SortedSpikesDecodingV1`, `ClusterlessDecodingV1`) | Anything that consumed the merge entry is stale — both because the spikes changed and because the half-length recording has different position/behavior coverage. |

In practice you would `(table & key).delete()` from the lowest stale row
upward, fix the selection, and re-`populate()`. DataJoint will block deletion
of parents until children are gone, so this is more or less forced on you.

## Can the old curation be reused?

**Not directly.** Three reasons, in order of severity:

1. **Unit IDs are not stable across sorter runs.** Even if you re-ran the same
   sorter with the same parameters on the same data, KiloSort / MountainSort do
   not guarantee identical unit numbering. Halving the recording will
   *definitely* change which clusters survive thresholds (some units present
   only in the dropped half disappear; firing rates and ISI distributions
   change; some clusters split or merge differently). A label like
   `unit_id=7 -> "accept"` from the old curation is meaningless against the new
   sort.

2. **Metrics are recomputed.** `nn_noise_overlap`, `snr`, `isi_violation`,
   `presence_ratio`, etc. are all functions of the spike train and waveform
   snippets on the new (shorter) recording. Anything you decided based on those
   metrics needs to be re-evaluated. Presence ratio and firing-rate-based
   thresholds are particularly sensitive to recording length.

3. **Manual curation in sortingview is bound to the sort.** A FigURL curation
   exports labels/merge groups keyed by the sort's unit_ids and is replayed
   against the same sorting object. There is no built-in "re-map an old
   curation onto a new sort" facility.

### What you *can* preserve

- **Curation parameters** — `MetricParameters`, `WaveformParameters`,
  `MetricCurationParameters`, sorter parameters (`SpikeSorterParameters`).
  Those are reusable as-is and are the cheap part to re-apply.
- **Your judgment / notes**, if you exported them. If you want to carry
  decisions forward, the practical move is to `fetch()` the old curation's
  labels and merge groups, hand-match units between the old and new sorts
  (e.g. by template similarity / waveform correlation on the overlapping time
  window), and then insert a new `CurationV1` row from those mappings. This is
  manual work; Spyglass doesn't do it for you.
- If the *only* reason you shortened the interval is to drop a noisy tail and
  the units in the kept window are unchanged, you'll often find the
  sort is *similar* but not identical — still treat it as a fresh curation.

## How to even do this safely

`SpikeSortingRecordingSelection` references `IntervalList` by
`(nwb_file_name, interval_list_name)`. You have two options:

1. **Make a new IntervalList entry** with a new `interval_list_name` (e.g.
   `..._first_half`) and **insert a new selection row** pointing at it.
   This is the recommended path — it preserves the original selection / sort
   /curation as a reproducible artifact, and the new pipeline runs alongside.
   You'll have two `SpikeSortingRecording` rows on disk, two sorts, two
   curations; the merge table can hold both. Comparing old vs new is then
   straightforward.

2. **Mutate the existing IntervalList row in place.** This is the path the
   question implies ("shorten the sort_interval on one row"). It's destructive:
   you lose the ability to reproduce the old recording and sort, and you must
   cascade-delete every downstream row first. DataJoint will not let you change
   a primary-key-referenced field on a populated parent without dropping the
   children. Avoid this unless you have a strong reason.

If you go with option 1, you do **not** lose the old curation at all — it just
lives against the original interval. Option 2 is when you "lose" it.

## TL;DR

- Everything from `SpikeSortingRecording` downward (`SpikeSorting`, all
  `CurationV1` rows for that sort, metric/figURL curations, `SpikeSortingOutput`
  merge entry, and any decoding / `SortedSpikesGroup` / `BurstPair` built on
  it) becomes stale and must be re-populated.
- The old curation's *decisions* don't auto-port: `unit_id`s aren't stable
  across sorts, and metrics depend on recording length. You can reuse the
  *parameters* but not the labels directly.
- Strongly prefer adding a new `IntervalList` + new selection row instead of
  editing the existing interval in place, so the original curation stays
  reproducible.
